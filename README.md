# MCAAP Onboarding — Stencil source

Clean, buildable [Stencil](https://stenciljs.com/) source for the `<mcaap-onboarding>`
guided-tour web component. This is the "lift-out" version of the live tour running
in the prototype (`/onboarding.js` + `/onboarding-tours.js`) — same states, same
behaviour, authored as a proper Stencil component you can compile and drop into any
app (React, Angular, Vue, or plain HTML).

## What it does

A workflow-aware onboarding tutorial:

- **Welcome modal** on first visit (persisted, dismissible, "don't show again").
- **Tour menu** — a launcher with two modes: **By role** (curated learning paths — Operations Lead, Collection Manager, Language Exploiter, Clearance Officer, Reports & Briefings, Triage Reviewer) and **All tutorials** (pick any). Roles are defined in `tours.ts` (`ROLES`); Getting Started is auto-pinned to the top of every role path.
- **Spotlight coach-marks** — dims the app, cuts a hole around a real element, and
  points a tooltip at it. Steps with no target fall back to a centered card.
- **Deep-linking** — a step can navigate the host app to the right page before it paints.
- **Progress** bar + step counter, **keyboard nav** (← → Enter / Esc), **completion screen**.
- **Persistence** in `localStorage` (completed / skipped / never-show).

## Install & build

```bash
npm install
npm start          # dev server + self-contained demo shell (src/index.html)
npm run build      # emits dist/ + loader/ (custom elements bundle)
```

`npm start` serves a **self-contained demo shell** (`src/index.html`) that mocks the
host app: it carries every `data-tour` anchor across stub pages, wires
`navigateHandler`, logs the analytics events, and exposes buttons to open the menu,
start any tour and run validation. It needs no real app — use it to develop, demo and
e2e-test the component in isolation.

`package.json` and `stencil.config.ts` here are minimal starters — merge them into
your project if you already have a Stencil setup.

## Testing

```bash
npm test              # spec tests (jsdom/mock DOM, no browser) — state machine,
                      # a11y wiring, analytics events, navigation seam, validation
npm run build         # emit www/ …
npm run test.smoke    # then: headless browser smoke test against the demo shell
```

`npm test` needs no browser and is CI-safe. `npm run test.smoke` drives the demo shell
in headless Chromium (`test/smoke.e2e.mjs`) to verify deep-linking, the spotlight, the
dialog a11y attributes, analytics and keyboard nav end-to-end; point it at a browser
with `CHROME_BIN=/path/to/chrome` if needed. Both suites are a foundation — migrate to
[`@stencil/vitest`](https://github.com/stenciljs/vitest) /
[`@stencil/playwright`](https://github.com/stenciljs/playwright) for a full CI suite
(Stencil's built-in `--spec`/`--e2e` runner is deprecated in v5).

## Use it

Load the component (from the built `loader`, or your framework bindings) and drop the
tag once, near the end of `<body>`:

```html
<mcaap-onboarding></mcaap-onboarding>
```

### Angular

A first-class Angular wrapper is generated by `@stencil/angular-output-target` on every
build — standalone proxy components that self-register the custom element (typed inputs,
methods and `EventEmitter` outputs). See [`angular/README.md`](./angular/README.md) for
build and usage. Add other framework bindings (React/Vue) the same way if needed.

### Trigger it

The component listens on `window` so any part of your app can drive it:

```js
// open the tour menu (e.g. from a Help "?" button)
window.dispatchEvent(new CustomEvent('mcaap-tour:open'));

// jump straight into a specific tour
window.dispatchEvent(new CustomEvent('mcaap-tour:start', { detail: { id: 'clearance' } }));
```

It also auto-shows the welcome modal ~0.9s after first load, unless the user has
completed it or ticked "don't show again".

### Analytics

The component emits typed DOM events for the tour funnel. They `bubble` and are
`composed`, so you can listen on `document` (or on the element / via framework
bindings) without a direct ref:

| Event                | `detail`                                     | Fired when |
|----------------------|----------------------------------------------|------------|
| `mcaapTourStarted`   | `{ id, title }`                              | a tour begins |
| `mcaapTourStep`      | `{ id, step, total, title }`                 | each step is entered (`step` is 1-based) |
| `mcaapTourCompleted` | `{ id, title }`                             | the last step is finished |
| `mcaapTourSkipped`   | `{ id, step, total }`                        | a tour is left before the completion screen (Esc, close, "Skip tour", or "All tours") |

```js
document.addEventListener('mcaapTourCompleted', e => {
  analytics.track('onboarding_tour_completed', e.detail); // { id, title }
});
```

### Wire up deep-linking + anchors

Two integration points connect the tour to your app:

1. **Navigation** — `tours.ts` steps carry a `page` hint. Point the component at your
   router by setting the `navigateHandler` property (the single integration seam):

   ```js
   const onb = document.querySelector('mcaap-onboarding');
   onb.navigateHandler = (page) => router.navigate(page); // page is the step's `page` string
   ```

   Alternatively, listen for the `mcaapNavigate` event (`{ detail: { page } }`) and
   route yourself. If neither is wired, the component falls back to the prototype
   globals (`window.__setPage` / `__openTasks` / `__goAgents`), so the UX prototype
   keeps working unchanged.

2. **Anchors** — spotlight steps target elements by CSS selector. Add
   `data-tour="..."` attributes to the elements you want to highlight (the prototype
   tags the header nav, the agent-fleet button, the help button, and the shared
   workspace header + scope tabs). If a target isn't found the step degrades
   gracefully to a centered card, so tours never break.

### Validate the wiring

Set the `debug` attribute (or call the `validate()` method) to check the tours against
the current DOM — it reports content problems, which spotlight anchors resolve on the
current page, which don't, and the distinct `page` hints you need to map:

```js
await document.querySelector('mcaap-onboarding').validate();
// { content: [], anchorsFound: [...], anchorsMissing: [...], pages: [...] }
```

Run it on each page of your app to confirm every `data-tour` anchor is present where
the tour expects it. (Against the demo shell, everything resolves once you navigate to
the matching page.)

## Editing content

All copy, ordering, illustrations and targets live in `tours.ts`. Add a tour by
pushing a `Tour` onto `TOURS`; add a step with `{ title, body, illus, target?, page?,
placement? }`. Illustrations are keyed strings in `illustrations.ts` — add your own or
swap in real `<img>`/GIF markup.

## Security

The component's trust model assumes tour content is **authored in-repo**:

- Step `title` / `body` are rendered as text nodes (auto-escaped by the JSX runtime) —
  safe to edit freely.
- The **only** `innerHTML` sinks are the icon (`ICONS`) and illustration (`ILLUS`)
  SVG strings, looked up by key from static maps in the source. `illus` on a step is
  a *key* into `ILLUS`, never raw markup.

If you ever source tour content from a CMS, API, or any user-controlled input, do
**not** feed it into `illustrations.ts` / `ICONS` (or any other `innerHTML`) without
sanitizing first (e.g. DOMPurify). Keeping illustrations authored in code avoids the
issue entirely. See the `TRUST BOUNDARY` comment in `mcaap-onboarding.tsx`.
